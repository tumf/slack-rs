# 変更提案: file download のリダイレクト追従と HTML エラー診断の強化

## 背景
Issue #19 の要件で `file download` の基本仕様は整理されていますが、運用時の失敗調査に直結する 2 点（リダイレクト追従の明示、HTML 応答時の診断情報）が現行要求で十分に明文化されていません。

## 目的
- Slack ファイル URL の一般的な挙動に合わせ、`file download` の HTTP 取得時にリダイレクト追従を必須化する。
- `Content-Type: text/html` を検知した失敗時に、原因特定に有用な短い本文断片（安全に切り詰めたスニペット）をエラー文脈として返せるように要求を明確化する。

## スコープ
- `wrapper-commands` の `file download` 要求を更新する。
- 実装・検証観点で CLI 出力と終了コードの期待値を追加する。

## 非スコープ
- 新しいサブコマンドの追加。
- 認証方式やトークン解決アルゴリズムの変更。
- 実ファイルのダウンロード性能最適化（並列化、再試行ポリシー追加など）。

## 期待効果
- `curl -L` 相当の期待動作との不一致を防止できる。
- 認証不備や URL 種別ミス時の一次切り分けが速くなる。
