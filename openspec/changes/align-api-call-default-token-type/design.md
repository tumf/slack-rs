# 設計: api call の既定トークン推測

## 方針
- `api call` のトークン解決は優先順位を維持する。
  1. `--token-type`
  2. `profile.default_token_type`
  3. トークンストアの存在確認による推測（User があれば User、なければ Bot）
- `SLACK_TOKEN` の優先順位は現状のまま維持する。

## 変更点
- `api call` の未指定時に、トークンストアの `exists()` を使って User token の有無を判定し、既定の token type を決める。
- 判定ロジックは小さな純関数に切り出し、ユニットテストで検証可能にする。

## 代替案
- `auth status` の表示を「設定値のみ」に変更する。
  - 既存の表示期待を変える影響があり、今回の課題の中心は `api call` 側の選択不整合であるため採用しない。

## 影響と互換性
- 既定トークンの推測が Bot から User に変わる可能性がある。
- 明示指定がある場合は既存挙動と同じため、互換性は最小限。
