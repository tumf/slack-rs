# 変更提案: プロファイル/トークン保存の基盤を整備

## 背景
複数アカウント対応を実現するため、プロファイル情報（非秘密）とトークン（秘密）の保存方式を明確にし、CLI全体の基盤として提供する必要がある。

## 目的
- プロファイルの永続化（非秘密情報）
- トークンの安全な保存（OS keyring）
- `--profile` 解決の安定性（`team_id:user_id` キー）

## 対象範囲
- `profiles.json` のスキーマと保存/読み込み
- keyring の保存/取得/削除
- プロファイル解決（`profile_name -> team_id,user_id`）

## 対象外
- OAuth 認証フロー
- Slack API 呼び出し
- CLI コマンドの実装（auth など）

## 依存関係
- 後続の認証/コマンド機能は本変更に依存

## リスク
- keyring 依存のテストが不安定になりやすい

## 成功条件
- プロファイル情報を読み書きできる
- keyring にトークンを保存/取得/削除できる
- `(team_id, user_id)` を安定キーとして解決できる
