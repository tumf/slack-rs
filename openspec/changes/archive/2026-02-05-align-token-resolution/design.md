## 目的
ラッパーコマンドと `api call` のトークン解決の一貫性を高め、`auth status` の既定トークン種別表示を正確にする。

## 設計方針
- **単一の解決ルール**: ラッパーコマンドは共通のトークン解決関数を通す。
- **環境変数優先**: `SLACK_TOKEN` が設定されている場合はそれを優先し、トークンストアに依存しない。
- **既存の優先順位は維持**: `--token-type` 指定 > `default_token_type` > 暗黙のフォールバックの順は維持する。
- **表示の正確性**: `auth status` は `default_token_type` が設定されている場合にそれを表示し、未設定時のみ従来の推測ロジックを使う。

## トークン解決の詳細
- `SLACK_TOKEN` が設定されている場合は、そのトークンを使用する。
- `SLACK_TOKEN` が未設定の場合のみ、トークンストアから指定のトークン種別を取得する。
- `--token-type` が指定されている場合は、指定された種別として扱う（表示上の `token_type` は指定値を使用）。

## 代替案
- `SLACK_TOKEN` をトークンストアのフォールバックに留める案もあるが、Issue #13 の期待（環境変数で確実に動くこと）を満たしにくい。

## 影響範囲
- `src/cli/mod.rs` にある API クライアント生成ロジック。
- `auth status` 表示ロジック。
